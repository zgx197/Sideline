# Facet 重构方案

> 本文档用于冻结 Facet 后续重构的目标、边界、阶段方案与验收标准。  
> 这次重构的核心目标不是只修复 Godot 编辑器中的 `.NET: Failed to unload assemblies`，而是把 Facet 从“验证期混合工具台”收敛成“编辑器工作台 + 运行时调试工具”两条清晰链路。

---

## 1. 背景

Facet 在前一阶段为了快速验证 UI、Lua 热重载、运行时诊断、日志观察等能力，把较多实验性质的工具堆在了 `Facet Main Panel` 中。  
这种做法在验证期有效，但随着功能增多，已经暴露出几个明显问题：

- 编辑器面板与运行时逻辑耦合过深。
- 编辑器插件、主场景脚本、Facet 运行时共享同一程序集热区，放大了 Godot C# 重载时的程序集卸载问题。
- `Facet Main Panel` 同时承担入口页、实验室、诊断台、控制台等多重职责，定义不清晰。
- 挂机窗口与地下城窗口已经具备运行时测试按钮，导致编辑器侧与运行时侧能力重复。
- 后续继续扩展 Facet 时，维护者难以快速判断某项功能应落在编辑器侧还是运行时侧。

因此，本次重构首先要冻结产品定义，再围绕这一定义推进代码和目录结构调整。

---

## 2. 重构目标

### 2.1 架构目标

- 明确拆分编辑器侧逻辑与运行时侧逻辑。
- 让 `Facet Main Panel` 回归为“编辑状态下的统一入口与工作台首页”。
- 把热重载实验、运行时快照、诊断观察等能力迁移到运行时调试工具。
- 收紧运行时全局入口和跨层依赖，减少长生命周期对象链。

### 2.2 工程目标

- 降低 Godot 编辑器 `Rebuild` 时程序集卸载失败的概率。
- 为未来拆分 `Facet.Editor`、`Facet.Runtime.Debug` 等边界留出稳定空间。
- 让目录结构、命名空间与职责划分更符合长期维护要求。

### 2.3 开发体验目标

- 让维护者能快速判断“某项能力属于编辑器工具还是运行时调试”。
- 让编辑器入口更轻、更清晰，避免继续堆叠实验功能。
- 让运行时验证回归真实窗口上下文，提高调试结果可信度。

### 2.4 稳定性目标

- 编辑器插件关闭后，能够完整释放其 UI 与事件链。
- 运行时调试工具不依赖编辑器主面板是否打开。
- 导出版不依赖编辑器插件，也能正确运行 Facet 运行时。

---

## 3. 核心结论

### 3.1 Facet Main Panel 的定义

重构后，`Facet Main Panel` 的定义为：

**Facet 框架在 Godot 编辑状态下的统一入口与编辑器工作台首页。**

它保留的职责应偏向：

- 框架入口
- 编辑期配置
- 资源与定义校验
- 文档与开发入口
- 轻量状态摘要

它不应继续承担：

- 运行时热重载实验
- 当前页面或指定页面的 round-trip 测试
- active runtimes 快照查看
- 运行时调试命令入口
- 结构化日志证据链分析工作台

### 3.2 Runtime Debug 的定义

运行时调试工具的定义为：

**Facet 在游戏运行状态下用于验证、诊断和观察运行时行为的工具集。**

它应承接：

- Lua 热重载测试
- 页面切换联动测试
- Projection / RedDot / Binding 观察
- 运行时快照与调试导出
- 面向挂机窗口和地下城窗口的真实上下文验证

### 3.3 先冻结职责边界，再考虑程序集拆分

当前优先级不是马上把工程拆成更多 `csproj`，而是先把“职责该落在什么层”讲清楚、收干净。  
只有职责稳定后，再做程序集拆分，收益才是正向的。

## 4. 目标结构

建议将 Facet 的后续结构整理为四层：

### 4.1 Facet.Core

定位：纯领域能力与通用抽象层。

职责：

- 不依赖 Godot 编辑器。
- 提供通用配置、服务容器、日志接口、运行时上下文等基础能力。
- 承载 editor/runtime 共用的轻量契约与 DTO。

### 4.2 Facet.Editor

定位：编辑器工作台与编辑期工具层。

职责：

- 提供 `Facet Main Panel`。
- 提供编辑期校验、导航、摘要、入口。
- 不直接依赖 `FacetHost.Instance`。
- 不直接操控运行时对象图。
- 尽量只通过稳定协议读取状态。

### 4.3 Facet.Runtime

定位：Facet 的 Godot 运行时宿主与页面运行层。

职责：

- 承载 `FacetHost`、运行时服务装配、页面加载、Lua、Projection、RedDot、布局与日志。
- 可独立运行，不依赖编辑器主面板是否打开。
- 在导出版与编辑器运行时中保持一致行为。

### 4.4 Facet.Runtime.Debug

定位：运行时调试与验证层。

职责：

- 承接热重载测试、运行时快照、观察器、调试命令与状态导出。
- 优先附着于真实运行窗口，而不是继续回流到编辑器主面板。
- 与 editor 之间只保留窄协议，不共享运行时内部实现。

---

## 5. 职责边界

### 5.1 Facet Main Panel 保留能力

- Facet 框架简介与当前接入状态摘要
- 编辑期资源与定义校验入口
- 文档入口、开发说明入口、工作流入口
- 轻量健康度摘要
- 打开日志目录、文档目录、关键资源目录等辅助入口

### 5.2 Facet Main Panel 移出能力

- 运行时 round-trip 测试按钮
- 挂机页 / 地下城页热重载联动按钮
- active runtimes 运行时树查看
- 运行时日志证据链分析
- 运行时热重载实验室
- 运行时布局实验室

### 5.3 Runtime Debug 承接能力

- 当前页热重载测试
- 指定页热重载测试
- 页面切换测试与状态输出
- Projection / RedDot / Binding 观察
- 运行时诊断快照
- 运行时日志摘要与问题定位

### 5.4 Editor 与 Runtime 的约束

- Editor 不直接引用运行时内部服务对象。
- Runtime 不依赖 Editor 面板生命周期。
- 跨边界通信优先使用文件协议或稳定 DTO。
- 所有边界协议都应有固定路径、固定字段和固定责任方。

---

## 6. 目录与命名空间整理方向

本次重构不要求一次性把所有程序集完全拆开，但建议先按逻辑边界整理目录与命名空间。

### 6.1 推荐目录方向

```text
godot/
├── addons/
│   └── facet-tools/                  # Facet.Editor
│       ├── FacetToolsPlugin.cs
│       ├── FacetMainScreen.cs
│       ├── Validation/
│       └── Workspace/
└── scripts/
    └── facet/
        ├── Runtime/                  # Facet.Runtime
        ├── RuntimeDebug/             # Facet.Runtime.Debug
        ├── Layout/
        ├── UI/
        └── Lua/
```

### 6.2 推荐命名空间方向

- `Sideline.Facet.Editor`
- `Sideline.Facet.Editor.Validation`
- `Sideline.Facet.Editor.Workspace`
- `Sideline.Facet.Runtime`
- `Sideline.Facet.Runtime.Debug`

---

## 7. Editor 与 Runtime 的通信方式

建议将编辑器侧与运行时侧的协作收敛为窄协议，而不是继续共享大量内部实现类型。

### 7.1 推荐方式

- `status` 文件：由运行时输出当前状态摘要
- `snapshot` 文件：由运行时输出只读快照
- `command` 文件：由编辑器或其他工具写入命令请求
- `result` 文件：由运行时回写执行结果

### 7.2 设计要求

- 路径固定
- 格式固定
- 字段稳定
- 写入方与读取方职责清晰
- 错误状态可追踪
- 即使 editor 不打开，runtime 也不受影响

### 7.3 禁止事项

- Editor 直接通过全局单例访问 Runtime 服务
- Editor 直接依赖 Runtime 内部对象图
- 让主面板承担“调试命令总线”的角色

---

## 8. 当前代码中的主要问题点

### 8.1 Main Panel 过重

当前 `FacetMainScreen` 曾同时承载日志查看、热重载实验、布局实验、运行时诊断和观察器入口，已经偏离“编辑器首页”的定位。

### 8.2 Editor 与 Runtime 过深耦合

编辑器主面板曾直接依赖多类运行时桥接和快照对象，导致插件天然进入运行时实现热区。

### 8.3 主场景与 Runtime 热区重叠

主场景直接挂载 `FacetHost`，而编辑器又频繁打开主场景，这会让主场景脚本与插件代码共同处在热重载敏感区域。

### 8.4 全局入口过宽

`FacetHost.Instance` 在验证期提高了开发速度，但也让依赖方向变得松散，生命周期边界不够清晰。

### 8.5 工具重复

挂机窗口、地下城窗口和编辑器主面板曾同时承载相似测试能力，带来职责重复和维护重复。

---

## 9. 分阶段实施方案

建议按以下五个阶段推进。

### 阶段 1：职责冻结

目标：先把定义定死，避免后续一边实现一边摇摆。

交付物：

- 一份能力盘点与最终归属清单
- 一份 `Facet Main Panel` 最终信息架构定义
- 一份 `Runtime Debug` 能力清单与首批落点定义
- 一组 editor/runtime 边界约束与禁止事项
- 一组进入阶段 2 的准入条件

完成标准：

- 现有能力都已有明确归属
- 不再保留“先临时放 Main Panel，后面再看”的模糊地带

### 阶段 2：编辑器主面板收口

目标：让 `Facet Main Panel` 回到编辑器首页定位。

任务：

- 移除运行时实验入口
- 保留文档、校验、工作区、摘要能力
- 拆出 `Validation/` 与 `Workspace/` 等辅助模块

完成标准：

- 主面板只保留编辑期职责
- 插件代码热区明显收紧

### 阶段 3：Runtime Debug 建立

目标：建立独立、统一的运行时调试入口。

任务：

- 建立 `RuntimeDebugPanel`
- 聚合运行时诊断、红点状态、日志摘要、热重载测试
- 将业务页中的重复调试 UI 收敛进 Runtime Debug

完成标准：

- 调试入口统一
- 业务页不再承载成片的调试 UI

### 阶段 4：运行时宿主边界收紧

目标：让 `FacetHost` 和页面运行时更符合 Godot 导出模型与长期维护要求。

任务：

- 统一 Lua 脚本入口
- 区分编辑器环境与导出环境的脚本源实现
- 收紧全局访问路径与长生命周期依赖

完成标准：

- 开发环境与导出环境行为一致
- 运行时边界更清晰

### 阶段 5：装配边界冻结

目标：明确当前阶段的程序集边界结论。

结论：

- 当前不新增 `Facet.Editor`、`Facet.Runtime.Debug` 独立 `csproj`
- 继续由 `Sideline.csproj` 承载 Godot 直接加载的 Facet 宿主脚本
- 但边界已经冻结为“文件级清晰、职责级清晰、文档级清晰”

完成标准：

- 能清楚说明为什么当前不拆新程序集
- 能说明未来进入正式拆分的准入条件

---

## 10. 测试建议

为了验证这套重构真正落地，后续测试应至少覆盖以下维度：

- 打开/关闭 `Facet Main Panel`，确认插件可按需创建与释放
- 在 Godot 中 `Rebuild` 多次，观察程序集卸载失败是否降低
- 从挂机页与地下城页进入 `Runtime Debug`，确认调试入口统一
- 验证 `Runtime Debug` 中的热重载、红点、日志和桥接状态是否正常
- 验证导出版与编辑器运行时都能正常初始化 `FacetHost`

---

## 11. 当前状态

截至当前阶段，已经落地的方向包括：

- 编辑器主面板进一步收口
- 运行时调试页建立并承接主要调试能力
- Lua 脚本源调整为更符合导出模型的结构
- 工作区与校验目录定义从主面板逻辑中拆出

后续如果继续推进，应优先沿本文档的阶段顺序执行，而不是再回到“把临时实验能力继续堆回主面板”的路径。
