# Lattice ECS Core 核心模块

> 面向确定性、帧同步与高性能数据布局的 ECS 核心设计说明。
>
> 设计参考：FrameSync、Unity DOTS、Bevy ECS、enTT。

---

## 目标定位

`lattice/Core` 关注的是 ECS 的底层骨架，而不是完整游戏业务层。
它的职责是为后续系统调度、网络同步、回放与上层业务逻辑提供稳定的数据组织方式。

当前这份文档重点回答三个问题：

1. 这个核心模块想解决什么问题
2. 我们为什么按当前顺序推进实现
3. 每一阶段的交付物和验证点是什么

---

## 设计原则

### 1. 先可用，再逐步拉高性能

Lattice 不追求第一天就把所有 Archetype、Query Builder、调度图和网络回滚一次性做满。
更合理的方式是先把最小可验证骨架立住，再逐层替换掉临时实现。

推荐推进顺序：

```text
迭代 1：实体 + 组件存储 + 最小 World
迭代 2：查询模型 + 系统调度
迭代 3：内存布局优化 + SIMD / Cache 优化
迭代 4：快照 / 回放 / 同步支撑
```

### 2. 确定性优先

任何设计都必须服务于“相同输入得到相同结果”。
这意味着：

- 数据结构必须可预测
- 更新顺序必须稳定
- 内存复用和实体回收必须能检测失效引用
- 快照、回放和同步需求要在底层提前预留边界

### 3. 模块边界清晰

核心层只负责：

- 实体标识与生命周期管理
- 组件存储与访问
- 查询和遍历
- 世界状态组织
- 与调度/快照交互所需的底层能力

核心层不负责：

- 具体玩法逻辑
- UI 或客户端表现
- 资源管理
- 网络协议细节

## 模块分层

Lattice ECS Core 可以理解为 5 个层次：

### 接口层 Interface

定义最基础的 ECS 语义：

- `EntityRef`
- `IComponent`
- `ISystem` 或后续调度接口

### 数据层 Data

负责真正保存世界状态：

- `EntityRegistry`
- `Storage<T>` / `ComponentStorage<T>`
- `Frame`
- `World`

### 查询层 Query

负责把“组件集合”组织成可遍历、可过滤的查询结果：

- 单组件查询
- 多组件联合查询
- 过滤器与构建器
- 高性能块式迭代器

### 调度层 Scheduling

负责系统执行顺序与阶段组织：

- `Update`
- `FixedUpdate`
- 后续渲染前同步点
- 命令缓冲与事件总线

### 应用层 Application

最终承载具体玩法系统：

- 角色属性
- 战斗结算
- 掉落逻辑
- 地图与事件系统

## 当前核心路线

### 阶段 1：最小 ECS 骨架

目标：先把实体、组件与世界状态组织跑通。

交付物：

- `EntityRef`
- `EntityRegistry`
- `Storage<T>` 或等价组件存储
- `Frame`
- `World`
- 最小遍历样例

验证点：

- 可以创建/销毁实体
- 可以安全增删组件
- 可以遍历指定组件集合
- 可以检测实体失效引用

### 阶段 2：查询与系统调度

目标：把“能存数据”推进到“能稳定驱动逻辑”。

交付物：

- 查询接口
- 多组件联合查询
- 系统注册和 Tick 驱动
- 简单调度分组

验证点：

- 查询结果顺序稳定
- 系统执行顺序稳定
- 遍历过程中禁止结构性修改或通过命令缓冲延迟修改

### 阶段 3：性能优化

目标：让查询和遍历方式开始真正贴近高性能 ECS。

交付物：

- 块式遍历器 `ComponentBlockIterator<T>`
- 预取优化版本 `PrefetchedBlockIterator<T>`
- 更紧凑的存储布局
- 必要的 SIMD / cache 优化

验证点：

- 顺序遍历性能明显优于逐元素散访问
- 在 DEBUG 模式下可检测非法修改
- 在 RELEASE 模式下不引入多余分支

### 阶段 4：快照、回放与同步基础设施

目标：为确定性帧同步和回放能力打地基。

交付物：

- Frame 快照
- 状态序列化骨架
- 回滚恢复入口
- 同步校验辅助信息

验证点：

- 同一帧状态可稳定导出
- 回滚后再模拟结果一致
- 快照边界不污染运行时逻辑层

## 关键设计点说明

### EntityRef 与版本号

Lattice 使用 `Index + Version` 的实体引用结构，而不是简单整数 ID。
这样做的目的是在槽位复用时能识别“旧引用”。

收益：

- 避免悬挂引用误指向新实体
- 保持对象池/空闲列表复用时的安全性
- 为快照与调试提供更清晰的状态判断

### 块式遍历器

`ComponentBlockIterator<T>` 的存在，不只是为了“遍历更快”，而是为了把 ECS 查询从“逐对象访问”切换到“按连续内存批处理”。

它适合：

- 顺序扫描大量组件
- SIMD 批量运算
- 预取优化
- 高缓存命中访问模式

它不适合：

- 高度随机访问
- 需要频繁结构修改的遍历过程

### DEBUG 版本校验

在开发期，遍历过程中修改存储结构通常是最难排查的问题之一。
因此在 DEBUG 模式中，迭代器应尽量校验存储版本；一旦检测到修改，立刻抛错。

这样做的价值是：

- 错误更早暴露
- 问题定位更准确
- 调试成本明显下降

## 当前状态结论

当前 `lattice/Core` 应继续坚持“骨架先行”的策略：

- 先稳住实体生命周期
- 再稳住组件存储与查询
- 再做块式遍历与性能优化
- 最后补快照、回放和同步支撑

这条路线的重点不在于一步到位，而在于每个阶段都能被独立验证，并且不会破坏下一阶段继续演进的空间。

后续如果继续推进，建议优先围绕以下三个方向展开：

1. `Storage<T>` 与 `Frame` 的职责进一步收敛
2. 查询模型与块式遍历的最小可用整合
3. 与快照系统之间的明确边界定义
