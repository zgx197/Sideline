## 1. 为什么需要定点数？

### 1.1 浮点数的问题

计算机存储小数通常使用 **浮点数**（float/double），但它有个致命问题：**不确定性**。

```csharp
// 同样的代码，不同电脑可能结果不同！
float a = 0.1f + 0.2f;
// a 可能是 0.30000001 或 0.29999999
```

**原因**：
- 浮点数使用二进制科学计数法存储
- 某些十进制小数（如 0.1）在二进制中是无限循环的
- 不同 CPU/GPU 的浮点运算单元（FPU）实现有细微差异
- 编译器优化可能改变运算顺序

### 1.2 帧同步为什么需要确定性？

在多人在线游戏中，**帧同步**要求所有玩家的游戏状态完全一致：

```
玩家A的电脑: 角色位置 = (10.5, 20.3)
玩家B的电脑: 角色位置 = (10.5, 20.3)  ← 必须完全相同！
```

如果因为浮点数差异导致：
```
玩家A: 位置 = 10.50000001
玩家B: 位置 = 10.49999999
↓
1000帧后
玩家A: 位置 = 105.2
玩家B: 位置 = 104.8  ← 严重不同步！
```

### 1.3 定点数的解决方案

定点数使用**整数运算**模拟小数，**完全确定**：

```csharp
// 定点数：用整数存储，运算结果完全一致
long a = 6554;  // 表示 0.1 (6554/65536)
long b = 13107; // 表示 0.2 (13107/65536)
long c = a + b; // = 19661，表示 0.300003（确定性！）
```

## 2. 定点数是什么？

### 2.1 核心思想

**定点数 = 整数 × 固定比例因子**

```
真实值 = 存储的整数 / 比例因子
```

**类比**：把 1.5 元存储为 150 分
```
真实值 = 150 分 / 100 = 1.5 元
```

### 2.2 定点数 vs 浮点数

| 特性 | 定点数 | 浮点数 |
|------|--------|--------|
| **存储** | 整数（int/long） | 特殊格式（IEEE 754） |
| **小数点位置** | 固定 | 浮动 |
| **运算速度** | 快（整数运算） | 较慢（浮点单元） |
| **确定性** | ✅ 完全确定 | ❌ 可能有差异 |
| **范围** | 固定 | 动态 |
| **精度** | 固定 | 动态 |

### 2.3 形象理解

想象一把**固定刻度的尺子**：

```
浮点数（游标卡尺）：
0-----1-----2-----3-----4-----5
  ↑小数点可动，精度变化

定点数（普通直尺，最小刻度 1mm）：
0---1---2---3---4---5 (单位：mm)
实际值 = 读数 / 10 (转成 cm)
  ↑小数点固定在第1位
```

## 3. Qm.n 格式详解

### 3.1 什么是 Qm.n？

**Qm.n** 是定点数的标准记法：
- **m**：整数部分的位数
- **n**：小数部分的位数
- **总位数 = m + n**（通常还要1位符号位）

**常见格式**：
- **Q16.16**：16位整数 + 16位小数 = 32位（int）
- **Q48.16**：48位整数 + 16位小数 = 64位（long）

### 3.2 Q16.16 格式详解

使用 32 位有符号整数存储：

```
二进制布局：[符号 1位][整数 15位][小数 16位]

示例：存储 1.5
1.5 × 65536 = 98304
二进制：0000 0001 . 1000 0000 0000 0000
         (整数1)   (小数0.5 = 32768/65536)
```

**比例因子（Scale）**：
```
比例因子 = 2^n = 2^16 = 65536

存储值 = 真实值 × 65536
真实值 = 存储值 / 65536
```

### 3.3 Q48.16 格式（Lattice 使用）

使用 64 位长整数存储：

```
二进制布局：[符号 1位][整数 47位][小数 16位]

比例因子 = 2^16 = 65536
存储值 = 真实值 × 65536
```

**为什么是 16 位小数？**
- 精度 = 1/65536 ≈ 0.000015
- 足够表示游戏世界的精细位置
- 乘法运算时只需右移 16 位（`>> 16`），速度快

### 3.4 转换示例

```csharp
// Q16.16 转换
float real = 3.14159f;
int stored = (int)(real * 65536);  // = 205887
float back = stored / 65536f;       // = 3.141586（近似）

// 常见值的存储
值      存储值 (Raw)
------------------
0       0
0.5     32768      (65536/2)
1       65536
1.5     98304      (65536 + 32768)
2       131072
π       205887     (3.14159 × 65536)
```

## 4. 基本运算原理

### 4.1 加法/减法

**最简单**：直接相加存储值

```
真实值: 1.5 + 2.5 = 4.0
存储值: 98304 + 163840 = 262144
验证: 262144 / 65536 = 4.0 ✅
```

```csharp
public static FP operator +(FP a, FP b)
{
    return new FP { RawValue = a.RawValue + b.RawValue };
}
```

### 4.2 乘法（关键！）

**不能直接相乘**，需要调整比例：

```
真实值: 1.5 × 2.0 = 3.0
存储值: 98304 × 131072 = 12884901888

问题：12884901888 / 65536 = 196608，但 3.0 应该是 196608？
等等，验证一下：196608 / 65536 = 3.0 ✅

但等等，12884901888 太大了！超出了 32 位范围！
```

**正确公式**：
```
结果 = (a × b) / 比例因子
     = (a.Raw × b.Raw) / 65536
```

```csharp
public static FP operator *(FP a, FP b)
{
    // 64位中间结果防止溢出
    long result = ((long)a.RawValue * b.RawValue) >> 16;
    return new FP { RawValue = (int)result };
}
```

**四舍五入 vs 截断**：

```csharp
// 截断（直接右移）
result = (a * b) >> 16;  // 向零取整

// 四舍五入（+0.5 后右移）
result = (a * b + 32768) >> 16;  // Lattice 使用这种方式
```

### 4.3 除法

与乘法相反，需要**先扩大**再除：

```
真实值: 3.0 / 2.0 = 1.5
存储值: (196608 << 16) / 131072 = 98304 ✅
```

```csharp
public static FP operator /(FP a, FP b)
{
    long result = ((long)a.RawValue << 16) / b.RawValue;
    return new FP { RawValue = (int)result };
}
```

### 4.4 与整数运算

整数可以看作 Qm.0（没有小数位）：

```csharp
// FP + int
public static FP operator +(FP a, int b)
{
    // int 左移 16 位转成 FP
    return new FP { RawValue = a.RawValue + (b << 16) };
}

// FP × int
public static FP operator *(FP a, int b)
{
    // 直接乘，不需要调整
    return new FP { RawValue = a.RawValue * b };
}
```

## 5. 精度与溢出

### 5.1 精度分析

**Q16.16 的精度**：
```
最小单位 = 1/65536 ≈ 0.000015

可以精确表示的数：
- 整数：全部精确
- 0.5 (32768/65536)：精确
- 0.25 (16384/65536)：精确
- 0.1 (6554/65536)：近似 (0.09999)
- π (205887/65536)：近似 (3.141586)
```

**精度足够吗？**

对于游戏开发：
- 位置精度 0.000015：在 1000 单位地图上，误差约 0.015 单位，肉眼不可见
- 角度精度：旋转时平滑度足够
- 速度/加速度：积累误差在可接受范围

### 5.2 溢出风险

**32位 Q16.16 的范围**：
```
最大正值：32767.9999
最小负值：-32768.0
```

**乘法溢出**：
```
20000 × 20000 = 400,000,000 (存储值)
实际 = 400,000,000 / 65536 ≈ 6103 ✅

但：30000 × 30000 = 900,000,000
900,000,000 >> 16 = 13732 (应该是 13732) ✅

等等，验证：30000 × 30000 = 900,000,000
900,000,000 / 65536 = 13732.91 ✅

问题：如果结果超过 32767？
40000 × 40000 = 1,600,000,000
1,600,000,000 >> 16 = 24414 (应该是 40000！)
```

**结论**：
- 乘法结果必须在小数点前 15 位能表示的范围内
- **安全范围**：约 ±32768

### 5.3 64位 Q48.16 的优势

使用 `long`（64位）：
```
理论范围：±140,737,488,355,328
安全乘法范围：±2,147,483,648

实际游戏中几乎不可能溢出！
```

## 6. 实际代码实现

### 6.1 最简 FP 实现

```csharp
public struct SimpleFP
{
    public const int SCALE = 65536;  // 2^16
    
    public int RawValue;
    
    // 构造
    public static SimpleFP FromRaw(int raw) => new() { RawValue = raw };
    public static SimpleFP FromFloat(float f) => new() { RawValue = (int)(f * SCALE) };
    
    // 转换
    public float ToFloat() => RawValue / (float)SCALE;
    
    // 运算
    public static SimpleFP operator +(SimpleFP a, SimpleFP b) => 
        FromRaw(a.RawValue + b.RawValue);
    
    public static SimpleFP operator -(SimpleFP a, SimpleFP b) => 
        FromRaw(a.RawValue - b.RawValue);
    
    public static SimpleFP operator *(SimpleFP a, SimpleFP b)
    {
        long temp = (long)a.RawValue * b.RawValue;
        return FromRaw((int)(temp >> 16));
    }
    
    public static SimpleFP operator /(SimpleFP a, SimpleFP b)
    {
        long temp = ((long)a.RawValue << 16) / b.RawValue;
        return FromRaw((int)temp);
    }
}
```

### 6.2 使用示例

```csharp
// 创建
var a = SimpleFP.FromFloat(1.5f);
var b = SimpleFP.FromFloat(2.5f);

// 运算
var c = a + b;  // = 4.0
var d = a * b;  // = 3.75
var e = b / a;  // = 1.666...

// 输出
Console.WriteLine($"a = {a.ToFloat()}");
Console.WriteLine($"a + b = {c.ToFloat()}");
```

## 7. 常见问题

### Q1: 定点数比浮点数慢吗？

**A**: 在现代 CPU 上：
- 加减法：定点数更快（纯整数运算）
- 乘法：定点数可能慢（64位中间结果），但差别不大
- 除法：都慢，定点数略慢
- **总体**：游戏逻辑中差别 < 5%，可忽略

### Q2: 为什么不用 double？精度更高

**A**: double 同样有确定性问题的风险（虽然比 float 小），且：
- 64位存储，与 Q48.16 相同
- 运算更慢
- 不能解决根本问题（跨平台一致性）

### Q3: 三角函数（sin/cos）怎么办？

**A**: 使用**查找表（LUT）**：
```csharp
// 预计算 sin 值
static int[] SinTable = new int[1024];

static void InitTable()
{
    for (int i = 0; i < 1024; i++)
    {
        double angle = (i / 1024.0) * Math.PI * 2;
        SinTable[i] = (int)(Math.Sin(angle) * 65536);
    }
}

public static FP Sin(FP angle)
{
    int index = (angle.RawValue % (65536 * 2 * PI)) * 1024 / (65536 * 2 * PI);
    return FP.FromRaw(SinTable[index]);
}
```

### Q4: 物理碰撞检测精度够吗？

**A**: 对于 2D 游戏完全足够：
- 位置精度 0.000015
- 即使高速运动（100单位/秒），每帧（1/60秒）移动 1.67 单位
- 碰撞检测通常用 AABB，精度要求不高

## 总结

| 概念 | 要点 |
|------|------|
| **定点数本质** | 整数存储，固定比例因子转换 |

| **溢出** | 64位版本（Q48.16）几乎不会溢出 |
