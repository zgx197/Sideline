## 1. 为什么需要定点数？

### 1.1 浮点数的问题

计算机存储小数通常使用 **浮点数**（float/double），但它有个致命问题：**不确定性**。

```csharp
// 同样的代码，不同电脑可能结果不同！
float a = 0.1f + 0.2f;
// a 可能是 0.30000001 或 0.29999999
```

**原因**：
- 浮点数使用二进制科学计数法存储
- 某些十进制小数（如 0.1）在二进制中是无限循环的
- 不同 CPU/GPU 的浮点运算单元（FPU）实现有细微差异
- 编译器优化可能改变运算顺序

### 1.2 帧同步为什么需要确定性？

在多人在线游戏中，**帧同步**要求所有玩家的游戏状态完全一致：

```
玩家A的电脑: 角色位置 = (10.5, 20.3)
玩家B的电脑: 角色位置 = (10.5, 20.3)  ← 必须完全相同！
```

如果因为浮点数差异导致：
```
玩家A: 位置 = 10.50000001
玩家B: 位置 = 10.49999999
↓
1000帧后
玩家A: 位置 = 105.2
玩家B: 位置 = 104.8  ← 严重不同步！
```

### 1.3 定点数的解决方案

定点数使用**整数运算**模拟小数，**完全确定**：

```csharp
// 定点数：用整数存储，运算结果完全一致
long a = 6554;  // 表示 0.1 (6554/65536)
long b = 13107; // 表示 0.2 (13107/65536)
long c = a + b; // = 19661，表示 0.300003（确定性！）
```

## 2. 定点数是什么？

### 2.1 核心思想

**定点数 = 整数 × 固定比例因子**

```
真实值 = 存储的整数 / 比例因子
```

**类比**：把 1.5 元存储为 150 分

| 概念 | 例子中的对应 | 数值 |
|------|-------------|------|
| **真实值** | 1.5 元 | 1.5 |
| **存储值** | 150 分 | 150 |
| **比例因子** | 1元 = 100分 | 100 |

**验证**：
```
存储值 = 真实值 × 比例因子
150 = 1.5 × 100 ✓

真实值 = 存储值 / 比例因子
1.5 = 150 / 100 ✓
```

### 2.2 定点数 vs 浮点数

| 特性 | 定点数 | 浮点数 |
|------|--------|--------|
| **存储** | 整数（int/long） | 特殊格式（IEEE 754） |
| **小数点位置** | 固定 | 浮动 |
| **运算速度** | 快（整数运算） | 较慢（浮点单元） |
| **确定性** | ✅ 完全确定 | ❌ 可能有差异 |
| **范围** | 固定 | 动态 |
| **精度** | 固定 | 动态 |

### 2.3 形象理解

想象一把**固定刻度的尺子**：

```
浮点数（游标卡尺）：
0-----1-----2-----3-----4-----5
  ↑小数点可动，精度变化

定点数（普通直尺，最小刻度 1mm）：
0---1---2---3---4---5 (单位：mm)
实际值 = 读数 / 10 (转成 cm)
  ↑小数点固定在第1位
```

**联系概念**：
- **最小刻度 1mm** = **精度**（能表示的最小单位）
- **读数 / 10** = **真实值 = 存储值 / 比例因子**
- **比例因子 10** = 10mm = 1cm

```
例子：读出 15mm，实际是 1.5cm
- 存储值 = 15（格数）
- 比例因子 = 10（10格=1cm）
- 真实值 = 15 / 10 = 1.5cm
- 精度 = 1格 = 0.1cm = 1mm
```

## 3. Qm.n 格式详解

### 3.1 什么是 Qm.n？

**Qm.n** 是定点数的标准记法：
- **m**：整数部分的位数
- **n**：小数部分的位数
- **总位数 = m + n**（通常还要1位符号位）

**⚠️ 注意**：在游戏开发领域，"Q16.16" 常被宽松地用来指代"使用16位小数精度的定点数"，而不严格限定整数位数。实际上：
- **Q16.16**（严格定义）：16位整数 + 16位小数 = 32位（int）
- **Q48.16**：48位整数 + 16位小数 = 64位（long）

**两者的小数部分都是16位**，区别在于整数部分的位数和总位数。

### 3.2 32位 vs 64位定点数对比

| 特性 | Q16.16 (32位) | Q48.16 (64位) |
|------|---------------|---------------|
| **整数位** | 15位 | 47位 |
| **小数位** | 16位 | 16位 |
| **总位数** | 32位 | 64位 |
| **数据类型** | int | long |
| **整数范围** | ±32768 | ±140万亿 |
| **安全乘法范围** | ±256 | ±32768 |

**小数精度相同**：都是 16 位，比例因子都是 65536。

### 3.3 Q48.16 格式详解（Lattice 使用）

使用 64 位长整数存储：

```
二进制布局：[符号 1位][整数 47位][小数 16位]
           ↑高48位↑    ↑低16位↑

示例：存储 1.5
1.5 × 65536 = 98304
二进制：0000...0001 . 1000 0000 0000 0000
        (整数部分1)   (小数部分0.5)
```

**比例因子（Scale）**：
```
比例因子 = 2^16 = 65536

存储值 = 真实值 × 65536
真实值 = 存储值 / 65536
```

---

### 3.4 为什么选择 Q48.16 设计？

#### 为什么小数部分是 16 位（比例因子 65536）？

**1. 精度足够**

**什么是精度？**

精度就是能够表示的**最小单位**，即存储值变化 1 时，真实值变化多少：

```
精度 = 1 / 比例因子 = 1 / 65536 ≈ 0.000015
```

**与公式的联系**：
```
存储值 0 → 真实值 = 0/65536 = 0
存储值 1 → 真实值 = 1/65536 ≈ 0.000015  ← 精度！
存储值 2 → 真实值 = 2/65536 ≈ 0.000030
存储值 3 → 真实值 = 3/65536 ≈ 0.000046
```

**这意味着**：
- Q48.16 无法表示 0.00001（小于精度的数）
- 最小的非零数是 0.000015
- 连续的两个数之间间隔 0.000015

**游戏场景中的意义**：
- 在 1000×1000 的地图上，误差约 0.015 单位
- 肉眼完全无法察觉（一个像素通常 > 1 单位）
- 比像素级精度（通常 1/96 英寸 ≈ 0.01）更高

**2. 运算效率**
```csharp
// 16 位小数可以用位运算优化
// 乘以 65536 = 左移 16 位
// 除以 65536 = 右移 16 位

a.RawValue = (b.RawValue * c.RawValue) >> 16;  // 快速！
```

**3. 行业标准**
- Unity DOTS (FP) 使用 16 位小数
- Photon Quantum 使用 16 位小数
- FrameSyncEngine 使用 16 位小数
- 大量实践证明 16 位是最佳平衡点

#### 为什么选择 64 位（Q48.16）而不是 32 位（Q16.16）？

**32 位 Q16.16 的问题**：
```
整数部分只有 15 位（+1位符号）
范围：±32768

乘法溢出示例：
256 × 256 = 65536（存储值）
65536 >> 16 = 1（实际应该是 256！）❌

安全乘法范围：只有 ±256
对于游戏开发来说太小了！
```

**64 位 Q48.16 的优势**：
```
整数部分 47 位（+1位符号）
范围：±140,737,488,355,328

256 × 256 = 65536
65536 >> 16 = 1 ✅（正确！）

安全乘法范围：±2,147,483,648
游戏中几乎不可能溢出！
```

**实际游戏场景对比**：

| 场景 | 32位 Q16.16 | 64位 Q48.16 |
|------|-------------|-------------|
| 地图大小 | 最大 32768×32768 | 几乎无限 |
| 速度计算 | 100 × 100 = 溢出 ❌ | 100 × 100 = 10000 ✅ |
| 物理碰撞 | 容易溢出 | 安全 |
| 中间计算 | 需要频繁检查溢出 | 可以放心计算 |

**结论**：
- **小数位 16 位**：精度与性能的最佳平衡
- **整数位 48 位**：足够大的范围，避免溢出
- **总 64 位**：现代 CPU 原生支持，效率极高

### 3.4 转换示例

**与元/分例子的对应关系**：

| 概念 | 元/分 例子 | Q48.16 定点数 |
|------|-----------|--------------|
| **真实值** | 1.5 元 | 1.5 |
| **存储值** | 150 分 | 98304 (raw) |
| **比例因子** | 100 | 65536 (2^16) |
| **计算** | 150 / 100 = 1.5 | 98304 / 65536 = 1.5 |

**为什么比例因子是 65536？**

```
Q48.16 表示小数点固定在从右数第 16 位：

二进制：0000 0000 0001 . 1000 0000 0000 0000
          整数部分(48位)   小数部分(16位)

小数部分 1000 0000 0000 0000 (二进制) = 32768 (十进制)

所以：
存储值 = 整数部分 × 65536 + 小数部分
       = 1 × 65536 + 32768
       = 98304

真实值 = 98304 / 65536 = 1.5
```

```csharp
// Q48.16 转换
float real = 3.14159f;
long stored = (long)(real * 65536);  // = 205887
float back = stored / 65536f;       // = 3.141586（近似）

// 常见值的存储
值      存储值 (Raw)
------------------
0       0
0.5     32768      (65536/2)
1       65536
1.5     98304      (65536 + 32768)
2       131072
π       205887     (3.14159 × 65536)
```

## 4. 基本运算原理

### 4.1 加法/减法

**最简单**：直接相加存储值

```
真实值: 1.5 + 2.5 = 4.0
存储值: 98304 + 163840 = 262144
验证: 262144 / 65536 = 4.0 ✅
```

```csharp
public static FP operator +(FP a, FP b)
{
    return new FP { RawValue = a.RawValue + b.RawValue };
}
```

### 4.2 乘法（关键！）

**不能直接相乘**，需要调整比例：

```
真实值: 1.5 × 2.0 = 3.0
存储值: 98304 × 131072 = 12884901888

问题：12884901888 / 65536 = 196608，但 3.0 应该是 196608？
等等，验证一下：196608 / 65536 = 3.0 ✅

但等等，12884901888 太大了！超出了 32 位范围！
```

**正确公式**：
```
结果 = (a × b) / 比例因子
     = (a.Raw × b.Raw) / 65536
```

```csharp
public static FP operator *(FP a, FP b)
{
    // 64位中间结果防止溢出
    long result = ((long)a.RawValue * b.RawValue) >> 16;
    return new FP { RawValue = (int)result };
}
```

#### 为什么需要右移16位？（比例因子叠加原理）

理解这个关键问题，要从**比例因子**的本质说起。

**回顾**：Q48.16 中，每个存储值都带有一个「隐含的」比例因子 65536：
```
存储值 = 实际值 × 65536
```

**直接相乘会发生什么？**

假设两个数的存储值分别是 `a_raw` 和 `b_raw`：
```
a_raw = 实际A × 65536
b_raw = 实际B × 65536

direct_result = a_raw × b_raw
              = (实际A × 65536) × (实际B × 65536)
              = (实际A × 实际B) × 65536 × 65536
              = (实际A × 实际B) × 65536²   ← 比例因子变成了 65536²！
```

**问题**：比例因子被乘了两次，变成了原来的平方！

**我们想要的结果**：
```
结果存储值 = (实际A × 实际B) × 65536   ← 只需要一个 65536
```

**修正方法**：多除以一个 65536（即右移 16 位）
```
(a_raw × b_raw) >> 16
= [(实际A × 实际B) × 65536²] / 65536
= (实际A × 实际B) × 65536   ✓ 正确！
```

**类比理解**：
> 就像你用「分」来存储金额（1元 = 100分）：
> - 1.5元存为 150 分（×100）
> - 2.0元存为 200 分（×100）
> - 直接相乘：150 × 200 = 30,000（分²）
> - 但正确结果应该是 3.0元 = 300 分
> - 所以需要多除以 100：30,000 ÷ 100 = 300 分 ✓

**结论**：定点数相乘时，**比例因子会叠加**，必须通过右移 16 位来消除多余的比例因子。

**四舍五入 vs 截断**：

```csharp
// 截断（直接右移）
result = (a * b) >> 16;  // 向零取整

// 四舍五入（+0.5 后右移）
result = (a * b + 32768) >> 16;  // Lattice 使用这种方式
```

### 4.3 除法

与乘法相反，需要**先扩大**再除：

```
真实值: 3.0 / 2.0 = 1.5
存储值: (196608 << 16) / 131072 = 98304 ✅
```

```csharp
public static FP operator /(FP a, FP b)
{
    long result = ((long)a.RawValue << 16) / b.RawValue;
    return new FP { RawValue = (int)result };
}
```

### 4.4 与整数运算

整数可以看作 Qm.0（没有小数位）：

```csharp
// FP + int
public static FP operator +(FP a, int b)
{
    // int 左移 16 位转成 FP
    return new FP { RawValue = a.RawValue + (b << 16) };
}

// FP × int
public static FP operator *(FP a, int b)
{
    // 直接乘，不需要调整
    return new FP { RawValue = a.RawValue * b };
}
```

## 5. 精度与溢出

### 5.1 精度分析

**Q48.16 的精度**：

回顾核心公式：
```
真实值 = 存储值 / 比例因子
       = 存储值 / 65536
```

**精度就是存储值变化 1 时的真实值变化**：
```
精度 = 1 / 65536 ≈ 0.000015

这意味着：
- 存储值从 0 → 1，真实值从 0 → 0.000015
- 存储值从 1 → 2，真实值从 0.000015 → 0.000030
- 相邻两个数的间隔永远是 0.000015
```

**可以精确表示的数**（分母是 65536 的倍数）：
```
- 整数：全部精确（如 1 = 65536/65536）
- 0.5 (32768/65536)：精确
- 0.25 (16384/65536)：精确
- 0.1 (6554/65536)：近似 (0.09999)，因为 0.1 不是 1/65536 的整数倍
- π (205887/65536)：近似 (3.141586)，因为 π 是无理数
```

**精度足够吗？**

对于游戏开发：
- 位置精度 0.000015：在 1000 单位地图上，误差约 0.015 单位，肉眼不可见
- 角度精度：旋转时平滑度足够
- 速度/加速度：积累误差在可接受范围

### 5.2 溢出风险

**32位 Q16.16（16位整数）的范围**：
```
int 范围：-2,147,483,648 到 2,147,483,647（约 ±21亿）
比例因子：65536
真实值范围：±32,768
```

#### 为什么乘法容易溢出？

以计算 **256 × 256** 为例：

**数学上**：256 × 256 = 65,536

**定点数计算过程**：
```
// 转换成存储值
a.Raw = 256 × 65536 = 16,777,216
b.Raw = 256 × 65536 = 16,777,216

// 乘法公式：(a.Raw × b.Raw) >> 16
中间结果 = 16,777,216 × 16,777,216
         = 281,474,976,710,656  （281万亿！）

// 右移16位：
281,474,976,710,656 >> 16 = 4,294,967,296
```

**问题**：结果 4,294,967,296 超过了 int 最大值（2,147,483,647）→ **溢出！**❌

#### 安全范围计算

为了保证乘法不溢出：
```
(a.Raw × b.Raw) >> 16 ≤ 2,147,483,647
即：a.Raw × b.Raw ≤ 140万亿

如果 a 和 b 的真实值都是 x：
(x × 65536) × (x × 65536) ≤ 140万亿
x² × 4,294,967,296 ≤ 140万亿
x² ≤ 32,768
x ≤ 181
```

**Q16.16 安全乘法范围：仅 ±181！**

#### 实际对比

| 格式 | 真实值范围 | 安全乘法范围 | 256×256 结果 |
|------|-----------|-------------|-------------|
| **Q16.16 (32位)** | ±32,768 | **±181** | 溢出 ❌ |
| **Q48.16 (64位)** | ±140万亿 | **±1100万** | 65,536 ✅ |

#### 游戏中的危险场景

```csharp
// 动能计算：0.5 × 质量 × 速度²
FP kineticEnergy = FP._0_50 * mass * velocity * velocity;

// 如果 velocity = 256
// velocity² = 65,536
// 在 Q16.16 中：velocity.Raw = 16,777,216
// velocity.Raw × velocity.Raw = 281万亿（溢出！）

// 在 Q48.16 中：
// 281万亿 >> 16 = 4,294,967,296（远小于 long 最大值 900万万亿）✅
```

**结论**：
- 游戏中的速度、位置、力等数值很容易超过 ±181
- Q16.16 会频繁溢出，导致错误结果
- **Q48.16 的安全范围（±1100万）在游戏中几乎不可能溢出**

### 5.3 64位 Q48.16 的优势

Lattice 使用 `long`（64位）：
```
理论范围：±140,737,488,355,328
安全乘法范围：±2,147,483,648

实际游戏中几乎不可能溢出！
```

**对比32位版本**：
- 32位 Q16.16：乘法很容易溢出（安全范围仅±256）
- 64位 Q48.16：即使大数字相乘也安全（安全范围±21亿）

#### 重要区分：业务安全 vs 算法安全

我们说 Q48.16 "不容易溢出"，这是**基于业务场景的工程结论**，而不是说它在算法层面有溢出保护。

| 层面 | 是否安全 | 说明 |
|-----|---------|------|
| **算法/数学层面** | ❌ 不安全 | Q48.16 底层就是普通 64 位整数，溢出照样截断 |
| **业务/工程层面** | ✅ 很安全 | 游戏数值远低于 140 万亿上限，实际不会触达 |

**算法层面的真相**：
```csharp
// 以下代码会溢出！Q48.16 不会阻止它
long max = 140737488355328L;  // Q48.16 最大值
long result = max + 1;        // 溢出！变成负数

// 以下乘法也会溢出
long a = 5000000000L;  // 50亿（远小于140万亿上限）
long b = 5000000000L;
long result = (a * b) >> 16;  // 中间结果 2.5e19 > 2^63，溢出！
```

**业务层面的安全**：
```
游戏设计决定：
    地图大小：1000 × 1000
    最大伤害：999,999
    最大金币：99,999,999
    
Q48.16 容量：±140,737,488,355,328（140万亿）

差距：10^5 ~ 10^7 倍的安全余量
```

**类比理解**：
> 就像你用**水库**装**一杯水**——水库的堤坝没有特殊加固（没有溢出保护机制），但你的水根本不可能漫出来。

**结论**：
- Q48.16 **不会**在算法层面防止溢出
- 它的"安全性"来自**容量设计的冗余**（140万亿 vs 游戏需要的几千几万）
- 正常游戏逻辑中几乎不可能触达上限，所以称为"事实安全"或"工程安全"

## 6. 实际代码实现

### 6.1 最简 FP 实现

```csharp
public struct SimpleFP
{
    public const int SCALE = 65536;  // 2^16
    
    public int RawValue;
    
    // 构造
    public static SimpleFP FromRaw(int raw) => new() { RawValue = raw };
    public static SimpleFP FromFloat(float f) => new() { RawValue = (int)(f * SCALE) };
    
    // 转换
    public float ToFloat() => RawValue / (float)SCALE;
    
    // 运算
    public static SimpleFP operator +(SimpleFP a, SimpleFP b) => 
        FromRaw(a.RawValue + b.RawValue);
    
    public static SimpleFP operator -(SimpleFP a, SimpleFP b) => 
        FromRaw(a.RawValue - b.RawValue);
    
    public static SimpleFP operator *(SimpleFP a, SimpleFP b)
    {
        long temp = (long)a.RawValue * b.RawValue;
        return FromRaw((int)(temp >> 16));
    }
    
    public static SimpleFP operator /(SimpleFP a, SimpleFP b)
    {
        long temp = ((long)a.RawValue << 16) / b.RawValue;
        return FromRaw((int)temp);
    }
}
```

### 6.2 使用示例

```csharp
// 创建

var b = SimpleFP.FromFloat(2.5f);

// 运算

// 输出

```

## 7. 常见问题

### Q1: 定点数比浮点数慢吗？

**A**: 在现代 CPU 上：
- 加减法：定点数更快（纯整数运算）
- 乘法：定点数可能慢（64位中间结果），但差别不大
- 除法：都慢，定点数略慢
- **总体**：游戏逻辑中差别 < 5%，可忽略

### Q2: 为什么不用 double？精度更高

**A**: double 同样有确定性问题的风险（虽然比 float 小），且：
- 64位存储，与 Q48.16 相同
- 运算更慢
- 不能解决根本问题（跨平台一致性）

### Q3: 三角函数（sin/cos）怎么办？

**A**: 使用**查找表（LUT）**：
```csharp
// 预计算 sin 值
static int[] SinTable = new int[1024];

static void InitTable()
{
    for (int i = 0; i < 1024; i++)
    {
        double angle = (i / 1024.0) * Math.PI * 2;
        SinTable[i] = (int)(Math.Sin(angle) * 65536);
    }
}

public static FP Sin(FP angle)
{
    int index = (angle.RawValue % (65536 * 2 * PI)) * 1024 / (65536 * 2 * PI);
    return FP.FromRaw(SinTable[index]);
}
```

### Q4: 物理碰撞检测精度够吗？

**A**: 对于 2D 游戏完全足够：
- 位置精度 0.000015
- 即使高速运动（100单位/秒），每帧（1/60秒）移动 1.67 单位
- 碰撞检测通常用 AABB，精度要求不高

## 总结

| 概念 | 要点 | 公式/数值 |
|------|------|----------|
| **定点数本质** | 整数存储，固定比例因子转换 | 存储值 = 真实值 × 比例因子 |
| **核心公式** | 真实值 = 存储值 / 比例因子 | 真实值 = 存储值 / 65536 |
| **比例因子** | 小数位的权重 | Q48.16: 2^16 = 65536 |
| **精度** | 最小可表示单位 = 1/比例因子 | 1/65536 ≈ 0.000015 |
| **Q48.16** | 48位整数 + 16位小数 | 64位 long 存储 |
| **加法** | 直接相加存储值 | a.Raw + b.Raw |
| **乘法** | 相乘后除以比例因子 | (a.Raw × b.Raw) / 65536 |
| **除法** | 先乘比例因子再相除 | (a.Raw × 65536) / b.Raw |

**概念联系图**：
```
真实值 ←────── 存储值 ──────→ 精度
    ↑            ↑              ↑
    └──────── 比例因子 ────────┘

真实值 = 存储值 / 比例因子
精度   = 1 / 比例因子
```

**统一公式对照**：
```
元/分例子：真实值(元) = 存储值(分) / 100，精度 = 0.01元
Q48.16：   真实值     = 存储值(raw) / 65536，精度 = 0.000015
```
