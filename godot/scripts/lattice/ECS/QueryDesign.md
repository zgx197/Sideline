# Lattice ECS Query 设计（Phase D3 已完成）

## 文档目的

本文档用于正式定义 Lattice 当前阶段的查询层设计。

这里的“Query”是一个概念层术语，表示：

- 系统如何从 `Frame` 中稳定读取一组匹配组件的数据

当前它**不是**一个统一的 `Query<T...>` 类型族，也不是完整的 query builder 系统。

本文档重点回答下面几个问题：

- 当前正式查询入口是什么
- `Filter<T...>()` 与 `ComponentBlockIterator<T>` 各自负责什么
- 默认路径与热点路径如何分层
- 未来的 owning group / zipped fast path 应该放在什么位置

---

## 当前定位

在当前底座链路中，查询层位于：

```text
Math
  ↓
Component
  ↓
Frame / Storage
  ↓
Query
  ↓
CommandBuffer / SystemScheduler
  ↓
SimulationSession
```

它的职责非常明确：

- 基于 `Frame / Storage` 提供稳定的数据读取入口
- 支撑系统在不直接操纵底层存储细节的前提下完成遍历

查询层当前不负责：

- 系统调度
- 结构修改
- 多帧管理
- 回滚策略

---

## 设计目标

### 1. 为普通系统提供稳定、低心智负担的默认入口

当前大多数业务系统都需要：

- 读取一个或多个组件
- 逐实体遍历
- 在遍历过程中直接修改当前组件值

因此查询层首先要解决的不是“表达能力最大化”，而是：

- 让普通系统能用稳定、直接、可预测的方式读到数据

### 2. 为热点系统保留更窄、更低层的遍历入口

并不是所有系统都适合走同一条路径。

有些热点系统更适合：

- 单组件 block 遍历
- 指针级访问
- 更少的抽象层

因此查询层必须保留：

- 正式默认路径
- 正式热点路径

而不是强迫所有系统都走完全同构的 API。

### 3. 当前不把未来的“统一 Query 抽象”写成事实

虽然从概念上说这是查询层，但当前主线已经落地并正式推荐的 API 是：

- `frame.Filter<T...>()`
- `frame.GetComponentBlockIterator<T>()`

所以本文档明确采用下面的写法：

- “查询层”
- “默认查询路径”
- “热点查询路径”

而不是把 `Query<T...>` 写成已经存在的正式公共 API。

---

## 当前正式入口

当前查询层正式入口只有两类。

## 1. 默认入口：`frame.Filter<T...>()`

当前默认推荐路径是：

- `frame.Filter<T>()`
- `frame.Filter<T1, T2>()`
- `frame.Filter<T1, T2, T3>()`

它们当前提供：

- 多组件匹配
- 自动主遍历源选择
- 零分配 `ref struct` 枚举器
- 与 `Frame` 的实体有效性检查协作

这条路径是当前普通系统的正式入口。

## 2. 热点入口：`frame.GetComponentBlockIterator<T>()`

当前热点路径是：

- `frame.GetComponentBlockIterator<T>()`
- `frame.GetComponentBlockIterator<T>(offset, count)`

它提供：

- 单组件 block 级遍历
- 实体引用与组件指针的直接返回
- 更低层的遍历控制
- 更适合做批处理和热点循环

这条路径当前不是默认路径，但已经是正式主线的一部分。

---

## 当前不属于正式主线的查询能力

下面这些能力当前不应写成默认查询层事实：

- `Query<T...>` 类型族
- 通用 query builder
- 自动查询计划器
- owning group 自动收敛
- zipped multi-storage fast path
- 基于表达式树的条件查询
- 完整并行 query abstraction

这些能力未来可能会存在，但当前阶段还不应倒逼底座文档和系统主线。

---

## `Filter<T...>()` 的正式语义

## 1. `Filter<T>()`

单组件 filter 的当前语义是：

- 以该组件存储为遍历源
- 逐项返回实体与组件指针
- 在遍历过程中结合 `Frame.IsValid(entity)` 过滤无效实体

因此它适用于：

- 单组件扫描
- 普通业务系统
- 不需要进一步手写 block 逻辑的场景

## 2. `Filter<T1, T2>()`

双组件 filter 的当前语义是：

- 自动选择元素数量更少的组件存储作为主遍历源
- 用另一组件存储做存在性校验
- 读取当前实体在两类组件中的指针

这条规则的关键价值是：

- 在不引入复杂查询计划器的前提下
- 用最小实现换取一个基本合理的主路径优化

## 3. `Filter<T1, T2, T3>()`

三组件 filter 的当前语义与双组件一致：

- 先在三个存储中选主遍历源
- 再对其余组件做存在性过滤
- 最后返回多组件引用

这说明当前 filter 模型的本质是：

- **以“选择最小主源 + 其他存储做 membership 校验”为核心策略的查询路径**

这已经足以支撑当前 system runtime 主线。

---

## `ComponentBlockIterator<T>` 的正式语义

## 1. 它是底层热点遍历器

`ComponentBlockIterator<T>` 当前的正式定位不是“更高级的查询器”，而是：

- 更低层
- 更靠近存储布局
- 更适合热点路径

它允许调用方：

- 逐个返回实体与组件指针
- 或者按 block 整块读取实体数组和组件数组

## 2. 它直接暴露存储布局优势

`Storage<T>` 当前采用 packed block 布局，因此 block iterator 能直接利用：

- 紧密排列的组件数组
- 顺序内存访问
- 更少的中间抽象

因此它适用于：

- 热点系统
- 批量循环
- 预取或 SIMD 友好的遍历

## 3. 它当前的边界

`ComponentBlockIterator<T>` 当前只天然适合：

- 单组件主源遍历

如果需要读更多组件，调用方需要：

- 额外用 `frame.GetPointer<TOther>(entity)` 补读
- 或者上层自己先判断是否值得走这条路径

这正是它与 `Filter<T...>()` 的角色边界：

- `Filter<T...>()` 优先表达“匹配多个组件”
- `BlockIterator` 优先表达“高效扫描一个主存储”

---

## 默认路径与热点路径的分层

当前查询层正式采用双路径分层，而不是单路径统治一切。

## 1. 默认路径

默认路径是：

- `frame.Filter<T...>()`

适合：

- 大多数业务系统
- 多组件扫描
- 需要较好可读性的系统代码
- 不在极致热点上的帧逻辑

当前建议：

- 普通系统优先使用 `Filter<T...>()`

## 2. 热点路径

热点路径是：

- `frame.GetComponentBlockIterator<T>()`

适合：

- 高频系统
- 单组件主遍历
- 对缓存命中、分支数、抽象成本更敏感的循环

当前建议：

- 只有在明确进入热点路径时，才退回 block iterator

## 3. 这两条路径的关系

它们不是替代关系，而是分层关系：

- `Filter<T...>()` 是默认主线
- `BlockIterator` 是正式保留的低层热点入口

后续所有文档与实现都应遵守这个分层，不要把热点路径写成默认写法。

---

## 查询层与 `Frame` 的关系

查询层当前不持有独立世界视图，它完全建立在 `Frame` 之上。

这意味着：

- 查询入口由 `Frame` 暴露
- `Filter<T...>()` 内部持有 `Frame`
- `BlockIterator` 从 `Frame` 取出底层存储
- 实体有效性最终仍由 `Frame.IsValid(entity)` 定义

因此查询层当前正式依赖方向是：

- `Query -> Frame / Storage`

而不是：

- `Frame` 反向依赖 query abstraction

---

## 查询层与 `Storage<T>` 的关系

当前 `Filter<T...>()` 和 `ComponentBlockIterator<T>` 都建立在 `Storage<T>` 之上。

这意味着：

- 查询层本质上是存储上的“受控访问层”
- 它不是独立索引系统
- 也不是二次缓存层

因此后续如需继续优化，优先级应是：

1. 优化存储访问路径
2. 优化主遍历源选择
3. 优化热点多组件组合方式

而不是先引入更复杂但脱离存储现实的 query abstraction。

---

## 查询层与系统的关系

当前 system runtime 对查询层的正式依赖方式是：

- 普通系统用 `Filter<T...>()`
- 热点系统可退到 `GetComponentBlockIterator<T>()`

这意味着查询层文档必须满足系统层需求：

- 写法足够直观
- 行为足够稳定
- 不引入隐藏调度副作用

当前系统文档中的推荐写法，后续都应以本文件为准继续收敛。

---

## 查询层与结构修改的关系

当前查询层只负责读取和就地修改当前已拿到的组件值。

它不负责：

- 结构性修改调度
- 延迟结构操作
- 临时实体映射

因此在系统遍历中如果需要：

- 创建实体
- 删除实体
- 添加组件
- 删除组件

应走 `CommandBuffer`，而不是试图直接把结构修改能力揉进 query abstraction。

这也是为什么 `ComponentBlockIterator<T>` 在 DEBUG 模式下会校验存储版本。

---

## 当前一致性约束

查询层后续实现都应遵守下面这些约束。

## 1. 查询必须建立在已注册组件之上

如果组件没有完成注册，就不应该进入查询主线。

这是因为查询最终仍然依赖：

- `Frame` 获取对应 `Storage<T>`
- 组件类型 ID 对应的存储定位

## 2. 查询不能重新定义实体有效性

查询层只能使用：

- `Frame.IsValid(entity)`

来判断实体是否仍有效。

它不应该自行维护第二套“实体是否还活着”的语义。

## 3. 查询层不应该屏蔽结构修改风险

当前迭代器在 DEBUG 下对版本变化直接报错，这是正确方向。

正式语义应该是：

- 边遍历边做结构性修改属于错误用法
- 应改用 `CommandBuffer`

## 4. 默认路径必须持续保持可读性优先

如果某个查询优化导致：

- 普通系统写法明显恶化
- 默认路径开始要求显式管理低层块细节

那么这类优化就不应该直接进入默认路径。

---

## `FullOwningGroup` 与未来热点路径的位置

当前代码库里已经存在：

- `FullOwningGroup`

它代表一种更激进的热点布局方向。

但当前正式文档结论是：

- `FullOwningGroup` 暂不属于默认查询主线
- 它也不应在 `D3` 阶段被写成当前通用 query 层事实

更准确的定位应是：

- 未来热点系统的显式 opt-in 布局方案
- 只有在真实 benchmark 证明值得收敛时才扩大使用面

同理，下面这些也属于后续优化预留：

- zipped fast path
- hot bundle
- owning group 注册维护

---

## 当前允许的实现方向

后续围绕查询层的实现，下面这些方向是允许的：

- 保持 `Filter<T...>()` 作为默认查询入口
- 保持 `ComponentBlockIterator<T>` 作为热点路径入口
- 继续优化多组件主遍历源选择
- 继续优化 block 遍历、预取与缓存友好性
- 在保持分层前提下，为热点系统引入显式 opt-in 的专用布局

---

## 当前不建议的方向

下面这些方向当前不建议进入主线：

- 把 `Query<T...>` 写成已经存在的正式 API
- 让所有系统默认下沉到 `BlockIterator`
- 为了未来并行而过早重写查询层
- 在没有真实收益前把 `FullOwningGroup` 推成通用默认路径
- 让查询层承担结构修改职责

这些都会让当前底座边界重新失焦。

---

## 与后续模块的关系

`QueryDesign` 对后续文档的影响如下：

- `CommandBufferDesign` 会建立在“查询只负责读取，不负责结构修改”之上
- `SystemDesign` 会建立在“普通系统走 Filter，热点系统可走 BlockIterator”之上
- `SimulationSessionDesign` 不应重新定义查询入口

如果后续文档与这些边界冲突，应优先回到本文件确认查询层职责。

---

## 后续收敛重点

当前 D3 完成后，查询层后续最值得继续收敛的方向是：

1. 明确双组件、三组件路径的持续优化边界
2. 评估哪些高频系统值得收敛到显式 owning group / hot bundle
3. 用 benchmark 验证默认路径与热点路径的真实差异
4. 在不破坏当前分层的前提下，再决定是否引入更正式的 Query 抽象

---

## 最终结论

Lattice 当前的查询层设计可以概括为：

**查询层当前由 `frame.Filter<T...>()` 和 `frame.GetComponentBlockIterator<T>()` 两条正式路径组成；前者是普通系统的默认入口，后者是热点系统的低层入口。**

这套双路径分层已经足以支撑当前 system runtime 主线，后续优化应建立在这套边界之上，而不是重新发明一套脱离 `Frame / Storage` 现实的抽象。
